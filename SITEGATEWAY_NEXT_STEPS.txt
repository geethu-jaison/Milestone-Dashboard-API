Site Gateway API (.NET 9) - Next Steps
=====================================

This document focuses on the concrete “what to create next” so the service follows your clean layered approach:
API controllers only in `Api` layer, orchestration/use-cases + interfaces in `Application`, and Dapper/data access + registry decrypt in `Infrastructure`, with DTOs/models in `Contracts`.

Repository (what you already have)
---------------------------------
You already have the solution and stub projects:
- `IconnectDashboardGateway.API`
- `IconnectDashboardGateway.Application`
- `IconnectDashboardGateway.Contracts`
- `IconnectDashboardGateway.Infrastructure`
- `IconnectDashboardGateway.Tests`

Step order (do this in sequence)
---------------------------------
1. Create the folder structure inside each project (see “Folder structure” section below).
2. Update project references (so each project can “see” the layer it should).
   - `Api` references `Application` + `Contracts`
   - `Infrastructure` references `Application` + `Contracts`
3. Add NuGet packages (Dapper + SQL + any auth/middleware packages you choose).
4. Define DTOs and shared request/response models in `Contracts`.
5. Define Application interfaces + use-cases (read-only endpoints first) in `Application`.
   - API controllers should depend only on Application interfaces/use-cases.
6. Implement Infrastructure:
   - registry decrypt + connection provider (K/D logic)
   - Dapper repositories that implement Application interfaces
7. Wire dependency injection:
   - `Application` has “abstractions” only; `Infrastructure` has concrete implementations
   - `Infrastructure` should expose an `AddInfrastructure(...)` extension method for DI
   - `Api` should call both `AddApplication(...)` and `AddInfrastructure(...)`
8. Implement API controllers for v1 routes (`/api/v1/...`), mapping HTTP <-> use-case calls.
9. Add Windows Service hosting (in `Api` Program.cs).
10. Add the first endpoint end-to-end (Health summary) before continuing to other summaries.

Dependency injection wiring (how “each layer talks”)
------------------------------------------------------
1) Application layer:
   - Define interfaces for repositories/providers (Application owns interfaces).
   - Define use-cases that take those interfaces in constructors.

2) Infrastructure layer:
   - Implement repository/provider interfaces.
   - Provide an extension method:
     - `AddInfrastructure(IServiceCollection services)`:
       - registers decrypted connection string provider
       - registers db connection factory (if you have one)
       - registers repositories as implementations of Application interfaces

3) API layer:
   - In `Program.cs`, call:
     - `services.AddControllers()`
     - `services.AddApplication()` (optional, if you create it)
     - `services.AddInfrastructure(...)`
   - Controllers should only:
     - validate input (route params/body)
     - call use-cases
     - return DTOs


Auth + security baseline (placeholders for now)
------------------------------------------------
Implement authentication early because it affects controllers’ behavior:
- API key authentication (header like `X-API-Key`), per-site credentials if required.
- Strict request validation for query parameters / route ids.
- Never log decrypted connection strings or registry secrets.

Recommended location:
- API layer middleware handles API key checks and rejects unauthorized requests.
- After auth passes, controllers call use-cases, which never handle secrets directly.


Initial endpoints (start read-only)
-----------------------------------
All endpoints should be under a consistent versioned route prefix:
- `/api/v1/...`

Suggested v1 route patterns:
- `/api/v1/sites/{siteId}/info`
- `/api/v1/sites/{siteId}/health/summary`
- `/api/v1/sites/{siteId}/cameras/summary`
- `/api/v1/sites/{siteId}/storage/summary`
- `/api/v1/sites/{siteId}/alerts/summary`
- `/api/v1/sites/{siteId}/license/summary`

Order to implement (to reduce risk):
1. Health summary (first end-to-end: controller -> use-case -> repository -> Dapper query)
2. Site info
3. Camera summary
4. Storage summary
5. Alert summary
6. License summary
